## 1. Kimi K3와 pelican benchmark에서 여전히 배울 수 있는 점

- **출처:** Simon Willison&#39;s Weblog
- **게시:** 2026.07.17 05:19
- **중요도:** 72
- **읽기 시간:** 6분

### 요약

Moonshot AI가 2.8조 개 파라미터의 Kimi K3를 발표했다. 현재 웹사이트와 API에서 사용할 수 있으며, 2026년 7월 27일까지 open weight 공개가 예정되어 있다. Kimi K3는 자체 보고 benchmark와 일부 외부 평가에서 높은 성능을 보였지만, 높은 추론 토큰 사용량과 비용도 확인됐다. 간단한 SVG 생성 실험에서는 13,241개의 reasoning token을 사용해 총 25센트가 들었고, 이미지 입력과 설명 생성은 양호했다.

### 왜 중요한가

Kimi K3는 대규모 모델의 성능뿐 아니라 토큰 사용량, 작업당 비용, reasoning 설정, vision 기능을 함께 평가해야 한다는 점을 보여준다. 또한 단일 이미지나 SVG benchmark는 모델의 기본 능력을 확인하는 데 유용하지만, 장시간 대화에서의 agentic tool calling과 도구 신뢰성까지 측정하지는 못한다.

### 개발자 영향

Kimi K3를 사용하면 입력 토큰당 3달러, 출력 토큰당 15달러의 비용과 높은 reasoning token 사용량을 고려해야 한다. 간단한 작업에서도 추론 비용이 커질 수 있으므로 모델 선택과 예산 산정에 영향을 준다. SVG 생성 같은 짧은 실험은 API 연결, 구조화된 출력, 공간 이해, vision 입력을 빠르게 점검할 수 있지만 실제 개발 워크플로의 장기 실행과 도구 호출 성능을 대신하지는 않는다.

### 권장 행동

Kimi K3를 검토한다면 동일한 간단한 프롬프트와 실제 개발 작업을 함께 실행해 출력 품질, reasoning token 사용량, 작업당 비용, 장시간 tool calling 안정성을 비교하라.

**키워드:** large language models, llm pricing, reasoning tokens, vision, svg generation, benchmarking, open weights, tool calling

## 2. Repository 관리자가 pull request를 보관할 수 있음

- **출처:** GitHub Changelog
- **게시:** 2026.07.17 04:01
- **중요도:** 58
- **읽기 시간:** 1분

### 요약

GitHub에서 Repository 관리자가 pull request를 영구 삭제하지 않고 public view에서 숨기는 archive 기능을 사용할 수 있게 되었습니다. 보관된 pull request는 closed 및 locked 상태가 되며, 개별 또는 일괄 처리할 수 있습니다. 관리자만 볼 수 있고 일반 방문자는 해당 URL에서 404 응답을 받습니다. 관리자가 unarchive하면 다시 표시되지만 closed 및 locked 상태는 유지됩니다.

### 왜 중요한가

spammy하거나 abusive한 pull request를 처리하면서도 기록을 보존할 수 있어, Repository moderation과 정책·법적 요구사항 대응에 유용합니다. 또한 is:archived 필터를 통해 보관 항목을 triage workflow에서 관리할 수 있습니다.

### 개발자 영향

Repository 관리자는 삭제 대신 접근 범위를 제한하는 선택지를 갖게 됩니다. 다만 보관된 pull request는 자동으로 closed 및 locked가 되며, 복원 후에도 이 상태가 유지되므로 review와 후속 논의가 필요한 항목을 보관할 때 상태 변경을 고려해야 합니다.

### 권장 행동

Repository의 moderation 및 보존 정책을 검토하고, 필요하다면 테스트용 pull request에서 개별 보관·복원과 is:archived 필터를 확인하세요.

**키워드:** github, pull requests, repository moderation, access control, triage

### 개발자 영향

### 권장 행동

GitHub Actions workflow의 \`runs-on\` 값을 지원되는 Xcode 27 레이블로 변경하고, arm64 제약과 설치된 도구 목록을 확인한 뒤 주요 빌드·테스트 파이프라인에서 검증하세요.

## 5. Thibault Sottiaux의 인용: Codex의 심각한 파일 삭제 버그

- **출처:** Simon Willison&#39;s Weblog
- **게시:** 2026.07.17 02:45
- **중요도:** 80
- **읽기 시간:** 1분

### 요약

Thibault Sottiaux에 따르면 GPT-5.6을 사용하는 Codex에서 파일이 예기치 않게 삭제되는 사례가 조사됐다. 이 문제는 Full access mode가 활성화되고 sandboxing 보호 및 auto review 없이 실행될 때 주로 발생하며, 모델이 임시 디렉터리를 지정하려고 $HOME 환경 변수를 덮어쓰다가 실수로 $HOME을 삭제하는 것으로 설명됐다.

### 왜 중요한가

AI 코딩 에이전트가 파일 시스템에 광범위한 권한을 가지면 모델의 단순한 경로 처리 실수가 사용자 파일 전체에 영향을 줄 수 있음을 보여준다. 따라서 에이전트의 생산성뿐 아니라 실행 권한과 검토 절차도 함께 고려해야 한다.

### 개발자 영향

Full access mode에서 sandboxing과 auto review를 사용하지 않으면 Codex의 작업이 치명적인 파일 삭제로 이어질 수 있다. 개발자는 파일 시스템 권한, $HOME 변경 동작, 자동 검토 여부를 워크플로의 주요 제약 조건으로 다뤄야 한다.

### 권장 행동

Codex를 실행할 때 Full access mode와 비샌드박스 실행을 피하고, 가능한 경우 sandboxing 보호와 auto review를 활성화한 뒤 파일 시스템 변경을 검토한다.

## 7. Inkling: Thinking Machines Lab의 오픈 웨이트 모델

- **출처:** Simon Willison&#39;s Weblog
- **게시:** 2026.07.17 00:35
- **중요도:** 78
- **읽기 시간:** 3분

### 요약

Thinking Machines Lab이 Apache-2.0 라이선스의 첫 오픈 웨이트 모델 Inkling을 공개했다. Inkling은 총 975B개 파라미터 중 41B를 활성화하는 Mixture-of-Experts 멀티모달 Transformer이며, 텍스트·이미지·오디오·비디오 45조 토큰으로 학습됐다. 276B개 파라미터와 12B 활성 파라미터를 사용하는 Inkling-Small도 개발 중이지만 가중치는 아직 공개되지 않았다.

### 왜 중요한가

Inkling은 최첨단 성능 자체보다 멀티모달 기능, 효율적인 추론, Apache-2.0 라이선스, Tinker를 통한 파인튜닝 가능성을 앞세운 기반 모델이다. 따라서 자체 데이터와 목적에 맞게 모델을 조정하려는 개발자에게 새로운 오픈 웨이트 선택지를 제공할 수 있다.

### 개발자 영향

개발팀은 Inkling을 Tinker에서 파인튜닝해 멀티모달 애플리케이션의 기반으로 평가할 수 있다. 다만 모델 카드와 학습 데이터 문서의 정보가 짧고 제한적이며, 학습 데이터에 공개 콘텐츠와 지식재산권 보호 대상일 수 있는 콘텐츠가 포함됐다고 명시되어 있으므로 라이선스·데이터 출처·운영 비용을 별도로 검토해야 한다. 또한 기사에 소개된 이미지 생성 및 이미지 설명 사례만으로는 실제 품질과 안정성을 판단하기 어렵다.
